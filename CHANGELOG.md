# Changelog

All notable changes to ACDP are recorded here. ACDP follows the versioning policy in [VERSIONING.md](VERSIONING.md).

## v0.1.0 — Clarifications addendum (round 3) — 2026-05-30

**Non-breaking specification clarifications and a new conformance fixture.** The RFCs remain `v0.1.0 Final`; this round records clarification/addendum items raised in an RFC follow-up review. No body field, schema `$id`, JCS rule, content-hash semantic, or signature semantic changed — every existing `v0.1.0` body, signature, and `content_hash` remains valid and wire-compatible. No JSON Schema changed.

- **Multi-tenancy implementation note (RFC-ACDP-0008 §6.4, new).** ACDP core defines no multi-tenant assignment — there is no protocol-level tenant identifier in the body, registry state, or capabilities, and no normative request-to-tenant mapping. Where a deployment partitions by tenant, the attribution MUST rest on an authenticated signal: implementations MUST NOT trust an unauthenticated tenant indicator (e.g. an `X-Tenant` header, hostname, or path segment) unless protected by deployment policy at a trust boundary, an authenticated gateway that stamps the tenant after authentication, or signed token claims. Treating an unauthenticated header as authoritative is a cross-tenant authorization bypass that the §4.5 visibility checks cannot catch, because they run *after* (forged) tenant attribution.
- **Cross-registry resolution is public-only in v0.1.0 (RFC-ACDP-0006 §4.4, new; §5 failure-mode row expanded).** v0.1.0 defines no mechanism to authenticate to a *remote* registry when following an `acdp://` reference — no bearer-token forwarding, token exchange, or delegated credential. A `restricted`/`private` context on a remote registry is therefore opaque (`not_found` / HTTP 404) to a cross-registry caller, who must authenticate to that registry directly, out of band. Authenticated remote retrieval (bearer-forwarding / token-exchange expectations and authority-trust rules) is out of scope for v0.1.0 and reserved for a future version (see RFC-ACDP-0009 §2.6).
- **Pagination MUST NOT terminate early on a fully-post-filtered page (RFC-ACDP-0005 §2.3).** Visibility scoping and other per-requester filters are applied after a storage page is read. A registry MUST NOT signal exhaustion (omit `next_cursor`) merely because a storage page contributed zero visible rows after post-filtering; it MUST advance the cursor to the next storage page while unscanned storage remains. A page MAY carry an empty `matches[]` together with a non-empty `next_cursor`. Early termination would silently truncate results and leak hidden-row boundaries.
- **Registry webhook event profile reserved (RFC-ACDP-0009 §2.10, new).** An OPTIONAL companion profile for registry-to-control-plane lifecycle events (`context_published` / `context_superseded` / `status_changed`) is reserved — distinct from §2.4 push subscriptions (which deliver search matches to discovery consumers). The reserved event shape carries `event_id`, `event_version`, `event_type`, `registry_authority`, `ctx_id`, `lineage_id`, `agent_id`, `visibility`, `derived_from`, and `created_at`; tenant binding is by deployment policy (consistent with §6.4), events SHOULD carry an HMAC-SHA-256 signature header (a registry-to-operator integrity check, NOT the producer signature), and delivery is at-least-once with `event_id`-keyed idempotent processing. Reserved profile name `acdp-registry-events`; not advertisable for conformance in v0.1.0.
- **RFC 8785 numeric conformance vectors (`can-011-jcs-numeric-vectors.json`, new; RFC-ACDP-0001 §5.2).** A new canonicalization fixture pins the RFC 8785 §3.2.2.3 (ECMAScript Number-to-String) behavior that JCS requires: exponential-notation boundaries (magnitude ≥ 1e21 → `e+`, ≤ 1e-7 → `e-`), the plain-decimal band, integer exactness through 2^53, negative-zero normalization, and IEEE 754 magnitude extremes — six executable vectors with byte-exact canonical forms and SHA-256 digests. §5.2's implementer note now cross-references it. Wired into the `acdp-registry-core` and `acdp-consumer` profiles (`registries/profiles.json`, `registries/profiles.md`) and the fixture index (`schemas/conformance/README.md`). The conformance runner executes it automatically; the suite now reports 30 arithmetic vectors.

## v0.1.0 — Clarifications addendum (round 2) — 2026-05-21

**Non-breaking specification clarifications and new conformance fixtures.** The RFCs remain `v0.1.0 Final`; this round tightens three SSRF / wire-convention points found during a second implementation review of the 2026-05-20 addendum. No body field, schema `$id`, JCS rule, content-hash semantic, or signature semantic changed — every existing `v0.1.0` body, signature, and `content_hash` remains valid and wire-compatible. No JSON Schema changed: the schemas already reject the inputs these fixtures pin; the gap was implementation behavior, not schema text.

- **DNS mixed-answer rejection made explicit (RFC-ACDP-0006 §7.1).** When DNS resolution returns multiple addresses and *any* of them is in a forbidden range, the implementation MUST reject the **entire** resolution for that hostname. Silently filtering out the forbidden addresses and connecting to the remaining safe ones ("filter and proceed") is now explicitly non-conformant — an attacker controlling both a public IP and a private target can return both, and a filter-based resolver reaches the private target on a retry, reconnect, or TOCTOU re-resolution. Applies identically to producer DID resolution (§4.8) and DataRef location fetches (§4.9). New fixtures `fed-007`, `did-ssrf-004`, `data-ref-ssrf-004` pin reject-the-whole-resolution behavior in all three fetch contexts.
- **Same-authority defined as scheme + host + effective port (RFC-ACDP-0006 §7.5).** The redirect "same authority" requirement now has a normative definition: two URLs share an authority only when scheme, host (case-insensitive), and effective port (after applying the scheme default — 443 for `https`) all match. An explicit `:443` on an `https://` URL is the same authority as the implicit default; `:8443` is not. A host-only comparison is non-conformant. The definition is cross-referenced from §4.8 and §4.9. New fixtures `fed-008`, `did-ssrf-005`, `data-ref-ssrf-005` pin same-host-different-port redirect rejection.
- **Absent-vs-null convention extended to `DataRef` and more wire types (RFC-ACDP-0002 §6.8, new).** A new subsection states that every `DataRef` optional field (`description`, `size_bytes`, `format`, `schema_version`, `content_hash`, `location`, `embedded`) follows the global absent-vs-null convention (RFC-ACDP-0005 §2.2.1): an unset field MUST be omitted, and an explicit `null` is a `schema_violation`. `location`/`embedded` being mutually exclusive does not make either nullable. New fixtures `schema-011`/`schema-012` pin `DataRef.format`/`DataRef.location` null rejection; `schema-013` pins `error.details`; `schema-014` pins `capabilities.limits.idempotency_key_ttl_seconds`.
- **Server-side authority validation (RFC-ACDP-0002 §3.1).** A registry MUST validate its own authority string before minting `ctx_id` or `origin_registry` values — it MUST be a bare DNS hostname (`acdp-common.schema.json#/$defs/hostname`), never a `host:port`, URL-form, uppercase, or DID-form authority. Libraries SHOULD enforce this at the constructor / configuration-load level so an invalid registry instance cannot be created.
- **Conformance manifest refresh (`registries/profiles.json`, `registries/profiles.md`, `schemas/conformance/README.md`).** The ten new fixtures are wired into the `acdp-registry-core`, `acdp-registry-federated`, and `acdp-consumer` profiles and the fixture index.

## v0.1.0 — Clarifications addendum — 2026-05-20

**Non-breaking specification clarifications and new conformance fixtures.** The RFCs remain `v0.1.0 Final`; this addendum adds normative clarifications and conformance fixtures discovered during implementation review. No body field, schema `$id`, JCS rule, content-hash semantic, or signature semantic changed — every existing `v0.1.0` body, signature, and `content_hash` remains valid and wire-compatible.

- **DataRef location SSRF protection (RFC-ACDP-0008 §4.9, new).** External `data_refs[].location` URLs are producer-controlled — the third attacker-controlled outbound fetch alongside producer DID resolution (§4.8) and cross-registry resolution (RFC-ACDP-0006 §7). A consumer dereferencing a `data_refs[].location` over HTTP(S) MUST apply the same SSRF posture: HTTPS-only, DNS-level IP-range filtering on every resolved address, IP pinning against rebinding, and a same-authority redirect cap. The defense is consumer fetch-time only — a registry does not fetch external `data_refs` and MUST NOT reject a publish on these grounds. New fixtures `data-ref-ssrf-001`/`-002`/`-003` pin IP-literal, DNS-rebinding, and cross-authority-redirect refusal; RFC-ACDP-0002 §6.5 now cross-references §4.9.
- **DNS-level SSRF protection made explicit (RFC-ACDP-0006 §7.1).** URL host-syntax checks are necessary but insufficient: a conformant implementation MUST resolve the hostname, validate every resolved address, and connect only to the validated address(es) without re-resolving. Calling a URL-syntax check and then using an HTTP client that resolves DNS normally is explicitly non-conformant (vulnerable to DNS rebinding and split-horizon DNS). Applies identically to §4.8 and §4.9.
- **Verification profile names (RFC-ACDP-0001 §9.2, new).** RECOMMENDED identifiers for SDK verification modes — `StrictV010` (the only v0.1.0-conformant mode, the §5.11 strict pipeline), `Diagnostic`, and `UnsafeForTests` — so documentation and error messages are consistent across implementations. `Diagnostic` and `UnsafeForTests` MUST NOT be exposed as default behavior. §5.11's strict-policy constructor name is corrected to `strict_v0_1_0()` (it previously read `strict_v0_0_1()`, a stale reference to a version number that never shipped — RFC-ACDP-0001 §1).
- **Consumer-profile SSRF requirement (RFC-ACDP-0001 §9.1, `profiles.json`).** The `acdp-consumer` profile's MUST list and behavioral requirements now state explicitly that consumers apply SSRF protections on every producer-controlled outbound fetch — producer `did:web` resolution (§4.8) and external `data_refs[].location` fetches (§4.9) — matching the fixtures the profile already requires.
- **Hostname vs host:port disambiguation (RFC-ACDP-0002 §3.1).** A normative table clarifying that `origin_registry` and the `ctx_id` authority MUST be a bare DNS hostname with no port, while `did:web` identifiers (`registry_did`, `agent_id`, `signature.key_id`) may carry a percent-encoded port — and that `did:web:<host>%3A<port>` is test/development-only.
- **`acdp_version` SDK default behavior (RFC-ACDP-0001 §6).** Builder-based SDKs SHOULD default to emitting `acdp_version: "0.1.0"` explicitly; an SDK taking the omission default MUST document it, provide a one-line explicit override, and state that absent and explicit `"0.1.0"` are interpreted identically though they are distinct `content_hash` preimages.
- **Conformance manifest refresh (`registries/profiles.json`, `registries/profiles.md`, `schemas/conformance/README.md`).** The three new `data-ref-ssrf-*` fixtures are wired into the `acdp-consumer` profile and the fixture index; a new `data-ref-ssrf` fixture family is registered.
- **Documentation refresh (`docs/`).** `threat-model.md` gains an SSRF row and attack scenario plus the previously-omitted supersession-race row; `integration-guide.md` sets `acdp_version` in the producer example and adds SSRF guidance for DID resolution, DataRef fetches, and cross-registry walks, plus a note that the consumer flow is the `StrictV010` profile; `architecture.md` notes SSRF protection on the cross-registry walk.
- **README / governance / manifesto editorial.** `README.md` repository map corrected — the `registries/` listing now includes `data-ref-types.md` and `profiles.json`, the `examples/` listing includes `key-resolution/`, and the `schemas/conformance/` listing reflects all current fixture families; the `acdp-consumer` profile row and the compatibility-model section are corrected (an inaccurate "embedding models" capability is removed). `governance/GOVERNANCE.md` and `governance/RFC-PROCESS.md` drop the stale "proto" references (ACDP is JSON-only), add `data-ref-types.md` to the registry list, reconcile the maintainers/core-team cross-reference, and add a security-vulnerability reporting path. `manifesto/manifesto.md` gains a closing pointer into the specification.

## v0.1.0 — 2026-05-19

**ACDP is promoted from Release Candidate to `Final`.** RFCs 0001–0008 now carry the `Final` status; RFC-ACDP-0009 remains `Reserved`. `0.1.0` is the first published Final version of the protocol — the `0.0.1` identifier named internal pre-release drafts only and was never promoted. `0.1.0` is wire-compatible with `0.1.0-rc1`: this release is a specification-hardening pass, not a wire change. No body field, schema `$id`, JCS rule, content-hash semantic, or signature semantic changed.

### Specification hardening since `0.1.0-rc1`

- **DataRef schema openness made explicit (RFC-ACDP-0002 §6.7, `acdp-data-ref.schema.json`).** The `DataRef` root object is now an explicit open schema (`additionalProperties: true`): root-level `DataRef` fields are producer-controlled and included in the `content_hash` preimage, so consumers MUST preserve unknown `DataRef` fields when recomputing the hash. The nested `embedded` sub-object remains closed. New fixture `can-010` pins the hash recomputation; the openness map in RFC-ACDP-0007 §3.3.1 gains a `DataRef` root row.
- **Absent-vs-null wire convention (RFC-ACDP-0005 §2.2.1).** Optional fields MUST be omitted, never serialized as JSON `null`, unless the schema explicitly types them nullable (`supersedes` is the one explicitly-nullable example). New fixtures `schema-005`/`schema-006`/`schema-007` pin rejection of `next_cursor: null`, `summary: null`, `domain: null`.
- **Producer DID resolution SSRF protection (RFC-ACDP-0008 §4.8).** Registries and consumers MUST apply the RFC-ACDP-0006 §7 SSRF defenses to producer `did:web` resolution, not only to cross-registry resolution. New fixtures `did-ssrf-001`/`did-ssrf-002`/`did-ssrf-003` pin loopback, IMDS, and private-range refusal; the `key_resolution_failed` code is broadened to cover SSRF-policy refusal.
- **Closed nested-schema conformance (RFC-ACDP-0007 §3.3.1).** New fixtures `schema-008`/`schema-009`/`schema-010` pin rejection of unknown fields in the closed `signature`, `data_period`, and `capabilities.limits` sub-objects.
- **`data_ref_hash_mismatch` SDK guidance (RFC-ACDP-0007 §5.3).** New guidance for SDK authors on keeping `data_ref_hash_mismatch` distinct from `hash_mismatch` and `invalid_signature`. New fixture `data-ref-008` pins the consumer-side external-DataRef mismatch case (body stays cryptographically valid).
- **Cursor conformance fixtures (RFC-ACDP-0005 §2.5.4).** New fixtures `cur-001` (`cursor_expired`) and `cur-002` (`invalid_cursor`).
- **`lineage_id` derivation golden vector (RFC-ACDP-0001 §5.6).** New dedicated fixture `lin-001`; the conformance runner now executes `lin-*` fixtures alongside `can-*`.
- **`acdp_version` producer guidance (RFC-ACDP-0001 §6).** Producers SHOULD set `acdp_version` explicitly; consumers, verifiers, and registries MUST NOT inject a default `acdp_version` into a body before recomputing `content_hash`.
- **Conformance manifest refresh (`registries/profiles.json`, `registries/profiles.md`, `schemas/conformance/README.md`).** All 14 new fixtures are wired into the `acdp-registry-core`, `acdp-registry-discovery`, and `acdp-consumer` profiles and the fixture index.
- **Editorial.** RFC headers, the RFC index, `README.md`, and `VERSIONING.md` updated from `Release Candidate 1` to `Final`. A `RELEASE.md` promotion checklist was added.

## v0.1.0-rc1 — 2026-05-18

**First published version of the Agent Context Distribution Protocol (ACDP), entering the world as Release Candidate 1.** RFCs 0001–0008 are at status `Release Candidate 1`; RFC-ACDP-0009 remains `Reserved`. The `0.0.1` identifier used by earlier pre-release drafts was never promoted to a Release Candidate or Final status and is superseded by `0.1.0`. `0.1.0` is wire-compatible with those drafts — the body format, JCS canonicalization, content-hash, and signature semantics are unchanged.

ACDP defines the minimal substrate for autonomous agents to publish, discover, and verify units of contextual information across distributed systems. It is JSON-only, coordination-agnostic, and has no central authority.

### Release Candidate scope

`0.1.0-rc1` is intended for implementation testing. Backward-incompatible changes remain possible until `Final`; only editorial fixes are expected during the RC window. Per [VERSIONING.md](VERSIONING.md), `0.1.0` is promoted to `Final` once the conformance suite — including the behavioral fixtures under `schemas/conformance/` — passes against at least two interoperating implementations.

### Normative RFCs (Release Candidate 1)

- **RFC-ACDP-0001 — Core.** Identifiers (`acdp://`, `lin:sha256:`), JCS canonicalization (RFC 8785), content hashing (`sha256:<hex>`), Ed25519 signatures, time format, `did:web` resolution, the v0.1.0 strict verification profile, and the **Body / ProducerContent / RegistryState** terminology that the rest of the spec depends on.
- **RFC-ACDP-0002 — Context Body.** The immutable signed body schema: context types, `data_refs` (URL / inline / embedded), visibility (`public` / `restricted` / `private`), `derived_from`, metadata bounds. `origin_registry` is normatively a DNS hostname, not a DID URI.
- **RFC-ACDP-0003 — Publish & Supersession.** `POST /contexts`, registry-assigned fields (`ctx_id`, `lineage_id`, `origin_registry`, `created_at`), the ordered publish-validation algorithm, supersession constraints, and `Idempotency-Key` semantics.
- **RFC-ACDP-0004 — Retrieval & Lineage.** `GET /contexts/{ctx_id}`, body-only retrieval, lineage queries (including `/current` semantics and lineage-endpoint visibility), visibility-aware cache headers, registry-attested status.
- **RFC-ACDP-0005 — Discovery.** Keyword search across the seven required fields, opaque cursors with ≥1h validity, requester-scoped pagination, `anonymous_public_reads` scoping, and whole-result-set `total_estimate`. Ranking is registry-defined.
- **RFC-ACDP-0006 — Cross-Registry References.** `acdp://` URI scheme, resolution flow, registry-DID-vs-producer-signature distinction, SSRF defenses.
- **RFC-ACDP-0007 — Capabilities & Errors.** `/.well-known/acdp.json` capability declaration, the standard error envelope, and the error-code registry (including `data_ref_hash_mismatch`).
- **RFC-ACDP-0008 — Security & Threat Model.** A2A threat model, required defenses, visibility enforcement, key-rotation rules, replay prevention.

### Reserved (numbering pinned, no normative text)

- **RFC-ACDP-0009 — Extensions.** Reserves the field namespaces and feature areas planned for future versions: retraction / lifecycle events, post-publication relationships, attestations, push subscriptions, server-side traversal (`/walk`), semantic similarity, cross-registry supersession, registry receipts.

### Conformance and tooling

- **11 canonical JSON Schemas** under `schemas/json/` covering context, body, registry-state, data-ref, publish request/response, search response, capabilities, the error envelope, common types, and the index. Schemas are namespaced under `schemas.acdp.io/v0.1.0/`.
- **Conformance fixtures** under `schemas/conformance/`: JCS canonicalization vectors, cryptographic golden vectors, and behavioral scenarios (publish, retrieval, lineage, visibility, capabilities, error-envelope, DataRef, metadata, body identity-field, and idempotency cases).
- **Conformance runner** (`scripts/conformance-runner.py`) that executes the arithmetic and cryptographic vectors end-to-end. Behavioral fixtures (`pub-*`, `vis-*`, `ret-*`, `body-*`, `err-*`, …) require a live registry to execute.
- **Open-vocabulary registries** under `registries/`: `auth-methods`, `context-types`, `error-codes`, `locator-schemes`, `media-types`, `profiles`, `signature-algorithms`. Each entry carries a status (`Proposed` / `Provisional` / `Stable` / `Deprecated`).
- **GitHub Actions CI** that validates JSON Schemas, examples, and conformance vectors on every PR.
- **`make bootstrap` + `make validate`** for one-shot local setup and verification.

### Conformance profiles

- `acdp-registry-core` *(default)* — RFC-0001..0004, 0007, 0008. Every conformant registry.
- `acdp-registry-discovery` — adds RFC-0005 keyword search.
- `acdp-registry-federated` — adds RFC-0006 cross-registry resolution.
- `acdp-consumer` — RFC-0001 + RFC-0004 (read) + RFC-0006.

### Examples

Working examples under `examples/` for capabilities, error envelopes, idempotency cycles, lineage walks, mixed `data_refs`, publishing, retrieval (including a real Ed25519 round-trip in `retrieval/golden-context.json`), keyword-search responses, supersession (v2 superseding v1), and visibility (`restricted` / `private` bodies). Every example is validated against the canonical schemas in CI.

### Hardening incorporated since the pre-release drafts

`0.1.0-rc1` folds in several rounds of specification hardening over the pre-release drafts, including: explicit lineage-endpoint visibility rules; `GET /lineages/{id}/current` semantics for all-superseded and expired-head lineages; `anonymous_public_reads` scoping of keyword search; whole-result-set `total_estimate`; the `data_ref_hash_mismatch` error code distinguishing DataRef-level integrity failures from body-level ones; a normative `origin_registry`-is-a-hostname rule with schema enforcement; and the v0.1.0 strict verification profile with recommended verification-report stage names.

### Explicitly out of scope in v0.1.0

Retraction (any form), post-publication `builds_on` relationships from third parties, attestations (`reproduced` / `disputes`), push subscriptions, server-side traversal, federation peering, cross-registry query forwarding, encrypted bodies, hard deletion, multi-party / threshold signatures, semantic similarity, and registry quality / reputation algorithms. Most of these are reserved in RFC-ACDP-0009 for future versions.
